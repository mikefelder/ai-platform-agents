| **1**         | **Partial Support – Low Customization**    | **Demonstrated in PoC.** POST /api/incidents captures title, description, severity, category, source, timestamp, and tags. | **2**         | **Partial Support – Medium Customization** | **Partially demonstrated.** Triage cascade applies severity-based logic: P1/P2 always require human approval, P3/P4 can auto-approve above 95% confidence threshold. APIM enforces rate limits and content safety per API. **Gap:** No formal versioned policy object applied before agent engagement. Policies exist as code (APIM XML, Python logic) but not as a first-class auditable entity with version tracking. Would need: policy registry service with version stamps, policy-applied event logged per incident.                                                                                                                                                                                                                     |
| **3**         | **Partial Support – Medium Customization** | **Partially demonstrated.** APIM acts as the AI Gateway for external traffic (frontend → APIM → agents). All 4 APIM APIs route through centralized policies with logging to Log Analytics → Sentinel. **Gap:** Internal agent-to-agent calls bypass APIM (UC2 calls UC1 directly container-to-container for latency; Bedrock tool calls AWS directly via httpx). To enforce zero bypass: route all inter-agent traffic through APIM (adds ~50ms latency per hop) and add network policies blocking direct container-to-container traffic. Medium effort.                                                                                                                                                                                       |
| **4**         | **Partial Support – High Customization**   | **Not demonstrated in current PoC.** Agents return responses and the FanInAggregator accepts them as-is. No data quality validation, gap detection, or contradiction analysis on agent outputs before they progress to synthesis. **To implement:** Add a validation executor between agent responses and the aggregator that checks completeness (required fields present), consistency (cross-reference conflicting claims), and flags gaps. Would require an LLM-based validation step or rule engine.                                                                                                                                                                                                                                      |
| **5**         | **Partial Support – Low Customization**    | **Partially demonstrated.** Bedrock tool has a 30-second polling timeout (_MAX_POLL_ATTEMPTS=30 × 1s). On timeout, returns an error message rather than silently continuing. OTEL spans capture the timeout with error=True. **Gap:** No formal SLA enforcement framework — timeouts are per-tool, not policy-driven. No automatic escalation to alternate agent or human on timeout. Low effort to add: configurable SLA thresholds per agent in agents.yaml, timeout event triggers workflow state transition to ESCALATED.                                                                                                                                                                                                                  |
| **6**         | **Partial Support – High Customization**   | **Not demonstrated.** No historical pattern matching or inconsistency detection against prior incidents. The PoC uses AI triage with rule-based fallback but doesn't compare diagnostic outputs against historical baselines. **To implement:** Would require a historical incident knowledge base (vector store or structured DB), a comparison agent that evaluates new diagnoses against past patterns, and confidence scoring on alignment. Significant build.                                                                                                                                                                                                                                                                             |
| **7**         | **Partial Support – Low Customization**    | **Demonstrated in PoC — strongest capability.** WorkflowBuilder.add_multi_selection_edge_group runs selected agents concurrently across Azure (Knowledge, Compliance, Governance) and AWS (Engineering/Bedrock). OTEL spans with custom attributes (agent.type=native/external, cloud.provider=aws) provide full cross-cloud visibility. W3C traceparent propagated to AWS Lambda. Grafana dashboard shows parallel execution timing. **Gap:** Current PoC runs Azure+AWS only (no OCI). Adding OCI would follow the same async MCP pattern used for AWS Bedrock — low customization to add a third cloud.                                                                                                                                     |
| **8**         | **Partial Support – Low Customization**    | **Demonstrated in PoC.** DecisionEngine evaluates agent votes with 3 strategies (weighted_majority, unanimous, quorum). Each AgentVote carries recommendation, confidence score, and reasoning. Decision object captures outcome, all votes, strategy used, confidence, and requires_approval flag. Full decision context stored on incident as attributes.decision. **Gap:** Remediation options aren't formally structured as risk/compliance profiles — votes are free-text recommendations. To formalize: add structured remediation option model with risk score, compliance tags, and estimated impact.                                                                                                                                  |
| **9**         | **Partial Support – Low Customization**    | **Demonstrated in PoC.** P1/P2 incidents always require human approval regardless of confidence. `POST /api/approvals/{id}/respond` accepts approve/reject with notes. Decision context (all votes, reasoning, confidence) is attached to the incident record. Timestamp recorded on approval. **Gap:** Approver identity not formally captured (no Entra auth on API). Approval linkage to incident is by ID but not cryptographically signed. Low effort: add Entra JWT validation middleware to capture approver UPN/OID.                                                                                                                                                                                                                   |
| **10**        | **Not Supported – In Roadmap**             | **Not demonstrated.** No real-time agent behavior monitoring, scope-based access control, or automatic agent suspension. APIM rate limiting prevents excessive calls but doesn't detect unauthorized scope access. **To implement:** Would require: (a) agent-level scope definitions (which systems/APIs each agent may access), (b) a runtime policy enforcement point that inspects every outbound call against allowed scope, (c) circuit breaker that suspends the agent on violation, (d) governance alert with plain-language summary. This is a significant platform capability requiring custom middleware or service mesh integration.                                                                                               |
| **11**        | **Partial Support – Medium Customization** | **Partially demonstrated.** Cosmos DB persists incident state with WorkflowState tracking current/previous status, agent results, decision confidence, and approval ID. State machine: received → triaging → investigating → deciding → awaiting_approval → remediating → resolved. WorkflowEvent log records every transition with actor, timestamp, and payload. **Gap:** True resumability after process restart is partial — in-memory caches (votes, workflow events) are lost on container restart. Cosmos persistence covers incidents but not all intermediate state. Real-time progress visibility requires polling APIs, not push (no WebSocket/SSE). Medium effort to fully persist all state to Cosmos and add progress streaming. |
| **12**        | **Partial Support – Low Customization**    | **Demonstrated in PoC.** APIM blocks 11 prompt injection/jailbreak patterns before agent processing. Azure AI Content Safety (DefaultV2) filters hate/violence/self-harm/sexual on every OpenAI call. OTEL telemetry flows to App Insights with custom span attributes (`uc`, agent.name, agent.type, tool.name). Grafana dashboard: token usage by model, latency by agent, cross-cloud metrics, tool executions. FinOps Workbook in Azure Monitor. **Gap:** Current PoC covers Azure+AWS (no OCI). Prompt/response monitoring is telemetry-based (after the fact), not inline real-time blocking of response content. Adding OCI telemetry follows same OTEL pattern.                                                                        |
| **13**        | **Partial Support – Medium Customization** | **Partially demonstrated.** All agent interactions emit OTEL spans to App Insights. APIM GatewayLogs flow to Log Analytics → Sentinel (5 analytics rules). Incident workflow events logged with actor/timestamp/payload. To implement: build a `/api/incidents/{id}/audit` endpoint that queries App Insights + Cosmos + workflow events and produces a structured JSON/PDF audit report. Medium effort.                                 |
